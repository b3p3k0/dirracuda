# ClamAV Integration Research Notes (2026-03-27)

Scope: validate current, primary-source behavior for ClamAV daemon/client tooling and choose a low-risk integration approach for Dirracuda.

## Key Findings

1. `clamd` is a long-running daemon and is the preferred high-throughput path for repeated scans.
- It listens on Unix and/or TCP sockets.
- The official docs warn that TCP control traffic is not authenticated and should not be exposed publicly.

2. `clamdscan` is a client for `clamd` and requires `clamd` to already be running.
- It accepts many `clamscan`-style flags but ignores most engine-config flags because scanning behavior is controlled by `clamd.conf`.
- Return codes are stable: `0` clean, `1` infected, `2` error.

3. `clamscan` does not require `clamd` but is slower for per-file loop usage.
- It initializes engine/database per invocation.
- Return codes are also `0` clean, `1` infected, `2` error.

4. Official docs advise against destructive auto-delete by default.
- `--remove` exists but is discouraged due to false positives.
- `--move`/`--copy` are safer patterns than delete.

5. For Python wrappers, package recency is mixed.
- `clamd` on PyPI is old (`1.0.2`, released 2014-08-21).
- `clamav-client` exists as a maintained fork and advertises support for both `clamd` and `clamscan`, but it still depends on ClamAV tooling/runtime.

6. Local environment snapshot on this machine (2026-03-27):
- `clamscan`: present
- `clamd`: missing
- `clamdscan`: missing

## Practical Implications For This Project

- Lowest operational risk for MVP: implement backend abstraction with `auto` selection:
  - prefer `clamdscan` when available
  - fallback to `clamscan`
  - if disabled or unavailable, keep current behavior unchanged
- Keep infected files in quarantine and promote only clean files to `~/.dirracuda/extracted`.
- Avoid hard dependency on old Python wrapper packages for first pass.
